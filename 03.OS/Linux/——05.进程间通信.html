        # 进程间通信（IPC）

        实际上在Linux发展历史上，进行进程间通信的方式有很多种。 在进程间完成数据传递需要借助操作系统提供特殊的方法，如：

        • 0、文件（弃用，开销大，稳定性低）
        • 1、管道、命名管道
        • 2、共享内存
        • 3、信号
        • 4、消息队列
        • 5、套接字等。

        随着计算机的蓬勃发展，一些方法由于自身设计缺陷被淘汰或者弃用。

        参数可能出现的错误

        思考：

        1. 可以open的时候O_CREAT一个新文件来创建映射区吗? 可以，但是不能创建出一个0字节大小的映射区。

        2. 如果open时O_RDONLY, mmap时PROT参数指定PROT_READ|PROT_WRITE会怎样？ 不行，权限不足，因为文件是只读，映射区却想读还想写

        3. 文件描述符先关闭，对mmap映射有没有影响？ 没有，因为文件描述符是用来操作文件的句柄的。

        4. 如果文件偏移量为1000会怎样？ 出现无效参数错误，必须是4K的整数倍。因为这个是内核里面的MMU帮你映射的、

        5. 对mem越界操作会怎样？ 不可以

        6. 如果mem++，munmap可否成功？ 不能成功，

        7. mmap什么情况下会调用失败？

        8. 如果不检测mmap的返回值，会怎样？ 很容易会出问题。。。除非你很有信心

        总结：使用mmap时务必注意以下事项：

        • 1.创建映射区的过程中，隐含着一次对映射文件的读操作。(实际上，要把文件读一遍)
        • 2.当MAP_SHARED时，要求：映射区的权限应 <=文件打开的权限(出于对映射区的保护)。而MAP_PRIVATE则无所谓，因为mmap中的权限是对内存的限制。
        • 3.映射区的释放与文件关闭无关。只要映射建立成功，文件可以立即关闭。(因为，操作文件的方式发生了变化)
        • 4.特别注意，当映射文件大小为0时，不能创建映射区。所以：用于映射的文件必须要有实际大小！！ mmap使用时常常会出现总线错误，通常是由于共享文件存储空间大小引起的。
        • 5.munmap传入的地址一定是mmap的返回地址。坚决杜绝指针++，'--'操作。
        • 6.如果文件偏移量必须为4K的整数倍
        • 7.mmap创建映射区出错概率非常高，一定要检查返回值，确保映射区建立成功再进行后续操作。

        # 3）没有血缘关系的进程之间mmmap通信

        实质上mmap是内核借助文件帮我们创建了一个映射区，多个进程之间利用该映射区完成数据传递。

        由于内核空间多进程共享，因此无血缘关系的进程间也可以使用mmap来完成通信。 只要设置相应的标志位参数flags即可。 若想实现共享，当然应该使用MAP_SHARED了。

        一个进程利用文件创建一个映射区，另外一个进程用相同的文件创建一个映射区。 然后把数据写到映射区里面。 关键：创建映射区的文件必须是相同的文件。

        直接用文件在进程之间进行通信也是可以的

        所以，实际上，早open文件之后，操作系统确确实实就是做了一个mmap

        # （2）信号的特征

        1.简单2.不能携带大量信息3.满足某个特设条件才发送。 所以，信号不能带个结构体过去。

        信号的原理和时钟中断的原理几乎是一样的，只不过时钟中断时通过硬件来完成的。 信号的话，是通过软件手段俩实现的。

        有的参考书，也称信号为软中断

        也就是借助软件手段实现的中断机制。 信号的特质：由于信号是通过软件方法实现，其实现手段导致信号有很强的延时性（这个说法是针对CPU说的）。但对于用户来说，这个延迟时间非常短，不易察觉。

        每个进程收到的所有信号，都是由内核负责发送的 内核处理。 所以严谨的讲，进程A给进程B发信号，这样的说法，应该说是A驱使内核给B发信号。

        # （3）如何产生信号？ 产生信号 5 种方式

        与信号相关的事件和状态

        产生信号 5 种方式：

        1. 按键产生，如：Ctrl+c、Ctrl+z、Ctrl+\ （比如Ctrl+Z 会导致，这个按键会导致内核向你当前执行的进程发送一个信号过去，这个信号会把进程杀死）

        5. 命令产生，如：kill命令，实际上其他命令也有，但一般研究这

        # （4）关于信号的状态和一些常用的名词。

        比如Ctrl+Z ，马上信号产生了-》马上递送-》递达 一般我们认为，从信号产生到最后递达，我们认为是瞬时的。可以说时间忽略不计。

        但是，信号产生之后，可能由于某种原因被阻塞(屏蔽)，这样就不能递达了，我们称之为“阻塞态”。Linux内核当中对这个阻塞态有一个描述的机制，这个描述他的机制对应的是一个集合：这个集合叫做“阻塞信号集”，还有另外一个名字，叫做“信号屏蔽字” 用此来描述信号的阻塞状态。

        与“信号屏蔽字”相呼应的，有另外一个集合，这个集合叫做：“未决信号集”.所谓未决，也就是这个信号没有被处理掉（注意理解这个，这个和前面）。所以说，他处于一个未决状态。也是在信号产生和递达之间，是**转瞬即逝（注意这个词语，虽然想吐槽，那设计这个干啥。。）**的

        递达：递送并且到达进程。 未决：产生和递达之间的状态。主要由于阻塞(屏蔽)导致该状态。

        # （5）PCB结构体中，有2个非常重要的成员：

        1）未决信号集-----为一个集合 2）信号屏蔽字（阻塞信号集）-----为一个集合 关系：“阻塞信号集”，影响（决定）“未决信号集” 当然：决定的前提是，信号要产生。 比如：我的某某进程，提前阻塞某编号的信号，0-》1 但是这个时候，那个编号的信号本来就没产生，他的未决是0，那么“阻塞信号集”就没影响。

        比如： 一个信号，要是不正常的，未决，那么未决信号集对应的编号从0-》1，然后处理，那么马上要从1-》0

        而，阻塞信号集，要是信号被阻塞，那么对应的编号从0-》1，而，阻塞过程，未决信号集对应的编号的0和1翻转，不能马上进行，所以，阻塞信号集，影响未决信号集

        2. 忽略(丢弃) （注意，这个是信号的处理方式，并不是说没有处理）
