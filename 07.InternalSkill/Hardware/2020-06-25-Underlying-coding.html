      # 原码，反码，补码——，移码

      # 一、实际背景

      现实生活中：在计算机中所有的数据运算都是采用补码进行的（比如Windows中） 注意：负数用补码表示，如果相加的和为负，也采用补码表示。

      # 二、有符号数的原码，反码，补码

      计算机中，有符号数有3种表示法：原码-->反码-->补码（发展历程，记得“简书”上有篇文章讲解了这个发展的历程，告诉了为什么一步步，最后用的补码） 但是现在目前的电脑其实都是补码表示了，当然，也有书上说：负数是补码表示，正数是原码/反码/补码表示，行吧，也算是对的。。谁让他们编码真的一样） （当然，个人从底层硬件实现来考虑，我觉得说正数是原码表示，负数是补码表示蛮好的，因为这样，我可以想象的是正数和负数首先全部编码为原码，然后根据符号位来判断，我要不要调用求补码的线路，正数不用，负数显然要） 记忆：符号位，0为正，1为负

      原码，反码，补码的正数表达是一样的

      # 1）原码：

      原码就是符号位加上真值的绝对值 既最高位为符号位，0为正，1为负，其余表示数值的大小 比如 正数+3=00000011 负数-3=10000011

      # 2）反码：

      原码，反码，补码的正数表达是一样的 正数+3=00000011

      负数的反码是在原码的基础上（符号位不变，其他位均取反） 负数-3=11111100

      # 3）补码

      原码，反码，补码的正数表达是一样的 正数+3=00000011

      负数的补码是在反码的基础上（符号位不变，在末位加1） 负数-3=11111101

      至此，我们可以总结出一种求一个负数的补码的规则： 写出负数的原码-》符号位不变，其他位取反（求反码）-》符号位不变，末位+1（求补码） 即：负数原码基础上，符号位不变，其他位取反，末位+1 上面，也是国内主流的教材上的方法（比如清华的那本计算机组成的教材）

      但是，我在另一些书上看到了下面两种求一个负数的补码方法：

      • 1）（唯一的,也是独特的）负数的原码基础上，直接变补码规则如下：符号位不变，从数的右边往左开始数,遇到“0”不理它,直到遇到第一个“1”为止，这个第一次遇到的1它本身不变,但事继续向左（直到符号位右边一个）的每一位取反 比如： 原码1001 0010 补码1110 1110
      • 2）先找到这个负数的正数编码，然后在这个基础上，所有位全取反，末位+1 比如：-1（我用8个比特说明） +1原码是0000 0001 全取反是1111 1110 末位加一是1111 1111 继续将，主流的方法补充到这，成为第3条
      • 3）负数原码基础上，符号位不变，其他位取反，末位+1 总结这三种方法：其实，本质都是等效的。。。我也不知道为什么，以前在自学的时候竟然看到了三种方法。

      Tips:对于负整数，补码的补码是原码（强调是负整数，有的书没有强调是负整数） 比如： 原码1001 补码（我用方法3））1001->1110->1111 原码（补码的补码，继续方法3）1111->1000->1001

      # 三、有符号数到无符号数的过渡标题

      重点： 计算机不区分有符号数和无符号数 只有有符号数有原码，反码，补码 计算机中存储数据 a.无符号数——直接以对应的二进制表示 b.有符号数——补码形式表示，无论是计算还是存储

      其实可以说无符号数没有原码，反码，补码一说，毕竟又不用分符号。

      《C陷阱与缺陷》一书在“整数溢出”一节中指出： 如果算术运算符的一个操作数是有符号数，另一个是无符号数，那么有符号数会被转换为无符号数。

      # 无符号数，竟然没有溢出一说?

      样例：

      #include<stdio.h>
int main()
{
	//4294967295对应二进制 11111111 11111111 11111111 1111111
	unsigned int a=4294967295;
	printf("无符号输出:%u\n",a);
	printf("有符号输出:%d\n",a);
	unsigned int b=1;
	printf("%u",a+b);
	return 0;
 }

      无符号的加法

      A11111111 11111111 11111111 11111111（表示32位的无符号最大值4294967295） A00000000 00000000 00000000 00000001（表示1） 1**00000000 00000000 00000000 00000000 丢弃最高位，所以等于0（硬件自动实现） 其实这种情况下个人认为是"溢出"了，但是可以通过下面的做法进行解释 //上面编译的，没有显示溢出的异常 解释：这样溢出后就相当于对（232-1）+1=232 ，再对和232 取模232得到为0

      但是，有本书上的说法，让我对“溢出”这个概念有些混乱 《C陷阱与缺陷》在“整数溢出”一节中指出，在无符号算术运算中，没有所谓的“溢出”一说。 有个博客对它的解释是： 我想Koenig的思路可能是这样，钟表上，当下午1点的时候，没有任何人会说：“现在是12点溢出了。” 因为我们已经常识性地知道，我们的钟表上是没有13这个数字的。

      虽然，这么说，个人还是觉得这种在一个数据类型边界出现较反常的，我都应该关注一下、 到目前为止我看过的好几本书的说法，真的有点弄晕我的感觉。但是，至此，我个人倾向是忘记什么叫溢出。只关注底层的比特的变化会不会造成影响。
